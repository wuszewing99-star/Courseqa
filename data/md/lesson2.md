# CourseQA 工程化

CourseQA 第10次课的目标是让系统有记忆、可追踪、更稳定。

有记忆指的是将用户 query 和系统返回的 hits 保存到 data/history.json 文件中。

可追踪指的是通过 logging 模块将服务启动、用户查询、检索结果和异常错误写入 logs/courseqa.log。

更稳定指的是在文件读写、JSON 解析和接口调用中加入异常处理，避免程序直接崩溃。

Python 项目开发不仅仅是编写几个函数，还需要考虑代码组织、文件结构、模块划分和运行方式。对于一个较完整的课程项目来说，通常会将核心业务逻辑、接口代码、工具函数、配置文件和数据文件分开放置。这样做的好处是每个文件职责明确，后续增加功能时不会把所有代码都堆在一个文件中。比如 CourseQA 项目中，检索逻辑可以放在 core 目录，接口逻辑可以放在 api 目录，日志和文件读写工具可以放在 utils 目录，配置项可以放在 config 目录。这种结构更接近真实软件项目，也有助于学生理解软件工程中的分层设计思想。

在 Python 项目中，配置文件的作用非常重要。项目中经常会出现一些固定参数，例如 Markdown 文档目录、历史记录文件路径、日志文件路径、默认返回结果数量以及文本截断长度等。如果这些参数分散写在不同代码文件中，后续修改会非常麻烦，也容易产生不一致的问题。将这些配置统一放入 settings.py 文件中，可以让项目更容易维护。例如当知识库目录从 data/md 改为 data/docs 时，只需要修改配置文件，而不需要在多个业务文件中反复查找路径字符串。这体现了工程开发中“集中管理配置”的思想。

Python 包的概念对于模块化项目非常关键。一个目录中如果包含 __init__.py 文件，通常可以被 Python 识别为包，从而支持 from package import module 这样的导入方式。虽然较新版本的 Python 对包的识别更加灵活，但在教学项目中保留 __init__.py 有助于学生理解包结构。比如 api、core、utils、stats 等目录都可以看作项目中的功能包。通过包结构，学生能够清楚地看到不同模块之间的关系，也能理解为什么不能随意把所有函数写在同一个文件中。

在 CourseQA 这样的问答系统中，核心检索逻辑应当保持相对独立。读取 Markdown 文档、切分文本块、构建 TF-IDF 索引、执行相似度计算，这些都属于核心业务逻辑。它们不应该直接处理前端页面，也不应该直接负责绘制图表。保持核心逻辑纯粹的好处是，后续无论是通过命令行调用、FastAPI 接口调用，还是前端页面调用，都可以复用同一套检索函数。这样的设计体现了软件工程中的“高内聚、低耦合”原则。

文本切分是构建问答系统时容易被忽略但很重要的环节。如果直接把一整篇 Markdown 文档作为一个检索单元，返回结果可能过长，也不够精确。更合理的做法是按照空行、标题或段落将文档拆分成多个文本块。每个文本块保留所属文件名、段落编号和具体文本内容。这样当用户查询某个知识点时，系统能够返回更精确的段落，而不是返回整篇文档。对于课堂演示来说，段落数量越丰富，检索结果越容易体现差异。

TF-IDF 是一种经典的文本检索方法，它通过词语在文档中的出现频率和在整体语料中的稀有程度来计算权重。一个词如果在某个文本块中频繁出现，但在其他文本块中较少出现，通常说明它对该文本块更有代表性。通过 TF-IDF 向量化之后，可以使用余弦相似度计算用户查询和文本块之间的相似程度。虽然 TF-IDF 不是最先进的语义检索方法，但它原理清晰、实现简单，非常适合作为课程项目中的基础检索算法。

余弦相似度常用于比较两个向量的方向是否接近。在文本检索中，用户查询和文档段落都可以转化为向量，然后通过余弦相似度计算它们的匹配程度。相似度越高，说明查询和文本块在关键词层面越接近。CourseQA 项目可以将所有段落的相似度排序，选出前 top_k 个结果返回给用户。通过调整 top_k 参数，用户可以控制返回结果数量，这也体现了接口参数设计对系统灵活性的影响。

在接口开发中，GET 和 POST 请求具有不同的使用场景。GET 请求通常用于获取数据，参数一般放在 URL 查询字符串中，例如 /query?text=Python&top_k=3。POST 请求通常用于提交较复杂的数据，参数可以放在 JSON 请求体中。对于问答系统来说，如果查询内容较短，GET 接口也可以使用；如果查询参数较多，或者需要传递结构化数据，那么 POST 接口会更加合适。学生需要理解 HTTP 方法不是随意选择的，而是与接口语义有关。

FastAPI 的一个重要优势是可以自动生成 Swagger UI 文档。只要定义好接口路径、请求参数和返回数据，FastAPI 就能在浏览器中展示交互式接口文档。学生可以直接在 Swagger UI 中填写参数并测试接口，不需要额外安装 Postman。对于教学来说，这极大降低了接口测试门槛。通过 Swagger UI，学生可以观察请求参数如何传递，返回结果如何展示，也可以理解接口文档在真实项目协作中的价值。

Pydantic 模型可以用于定义请求体的数据结构。例如在查询接口中，可以定义一个 QueryRequest 模型，其中包含 query 和 top_k 字段。FastAPI 会根据模型自动校验请求数据类型，如果用户传入的 top_k 不是整数，系统会自动返回参数错误。这种自动校验机制可以减少手动判断代码，提高接口稳定性。对于软件工程专业学生来说，理解数据模型和接口参数校验，是掌握后端开发的重要一步。

历史记录功能可以让系统保存用户每一次查询行为。一个完整的历史记录通常包括查询时间、用户问题、返回数量、命中文档、命中文本和相似度分数。保存历史记录的意义不仅是方便用户回看，还可以为后续统计分析提供数据来源。比如可以统计每天有多少次查询、哪些问题被重复提问、哪些知识点最受关注。这样，问答系统就不再只是一个检索工具，而变成了一个可以积累使用数据的教学辅助系统。

历史记录的格式设计非常重要。如果每次查询保存的数据结构不统一，后续统计和可视化就会非常麻烦。例如有的记录使用 query 字段，有的记录使用 question 字段；有的记录使用 time 字段，有的记录使用 timestamp 字段；有的记录包含 hits，有的记录没有 hits。这样的混合格式会导致 pandas 读取时出现缺失字段或空值问题。因此在项目升级时，应尽量保持历史数据结构稳定，新增字段可以兼容，但不应随意改变原有格式。

JSON 文件适合作为轻量级数据存储方式，特别适合课堂项目和小型原型系统。Python 提供 json 模块，可以方便地将字典和列表保存为 JSON 文件，也可以从 JSON 文件读取数据。保存 JSON 时应注意 ensure_ascii=False，这样中文内容可以正常显示；同时使用 indent 参数可以让文件格式更美观，便于人工查看。虽然 JSON 不适合大型高并发系统，但对于教学项目来说，它直观、简单，能帮助学生理解数据持久化的基本概念。

异常处理是保证系统稳定运行的重要机制。在文件读取、JSON 解析、接口调用和数据写入过程中，都可能出现错误。例如文件不存在、JSON 格式损坏、传入参数为空、索引尚未加载等。如果没有异常处理，程序可能直接崩溃，用户只能看到服务器错误。通过 try...except 捕获异常，并结合日志记录错误信息，可以让系统在出现问题时返回更友好的提示，同时方便开发者定位问题。这也是从脚本代码走向工程项目的重要一步。

日志系统可以记录程序运行过程中的关键信息。比如服务何时启动、读取了多少个 Markdown 文档、切分了多少个文本块、TF-IDF 索引是否构建成功、用户查询了什么问题、返回了多少条结果、是否发生异常等。日志不仅能帮助调试，也能作为系统运行状态的证据。在真实项目中，日志通常比 print 更专业，因为它可以设置不同级别，如 DEBUG、INFO、WARNING、ERROR，并可以输出到文件中长期保存。

pandas 是 Python 数据分析中非常常用的库，它提供了 DataFrame 表格结构，适合处理结构化数据。对于 CourseQA 项目来说，可以将 history.json 读取为 DataFrame，然后按日期分组统计每日提问量，按 query 字段统计高频问题，按 tag 字段统计科目分布。pandas 的优势在于语法简洁，groupby、value_counts、reset_index 等方法可以快速完成统计任务。通过 pandas，学生可以把项目中的业务数据转化为可分析的数据表。

Plotly 是一个支持交互式图表的 Python 可视化库。与普通静态图片不同，Plotly 生成的图表可以在浏览器中缩放、悬停查看数值、保存图片，也可以嵌入网页中。在 CourseQA 项目中，可以用柱状图展示每日提问量，用水平条形图展示 Top10 高频问题，用饼图或环形图展示科目分布。Plotly 的交互效果适合课堂演示，也能让学生感受到数据分析结果如何转化为直观的可视化表达。

在没有前端页面的情况下，运行 stats_visual.py 可以直接弹出图表窗口，这适合课堂测试和本地演示。通过 python -m stats.stats_visual 命令，程序会读取 history.json，完成统计计算，并依次弹出每日提问量柱状图、Top10 高频问题条形图和科目分类饼图。不过这种方式只适合开发阶段，因为它依赖本地浏览器弹窗。正式项目中，如果已经有前端页面，就不应该使用 fig.show，而应该通过接口返回图表数据或统计数据，由前端负责渲染。

如果项目有前端，那么后端接口就是前端和业务逻辑之间的桥梁。前端页面可以调用 /query 接口完成问答查询，也可以调用 /history 接口展示历史记录，还可以调用 /stats 接口获取统计数据。对于图表展示，后端可以直接返回统计数据，让前端使用 Plotly.js 或 ECharts 绘图。这样前后端职责更加清晰：后端负责数据处理和接口输出，前端负责页面布局和图表展示。这是 Web 项目中常见的前后端协作方式。

统计接口的设计应当尽量返回结构清晰的 JSON 数据。比如 total_questions 表示总查询次数，daily_counts 表示每日提问量，top10_questions 表示高频问题列表，subject_counts 表示科目分类统计。前端拿到这些数据后，可以自由选择展示方式，例如表格、柱状图、折线图、饼图或仪表盘。相比后端直接返回图片，返回结构化数据更加灵活，也更符合前后端分离项目的设计思想。

查询参数增强可以提升系统的可配置性。例如 min_score 可以过滤相似度过低的检索结果，max_chars 可以控制返回文本长度，top_k 可以控制返回条数。这些参数能够让同一个查询接口适应不同使用场景。比如课堂演示时可以返回较短文本，方便观察；实际使用时可以返回更多段落，提升信息覆盖率。参数设计不仅影响用户体验，也体现了接口设计的灵活性和扩展性。

测试是软件项目不可缺少的环节。对于 CourseQA 项目来说，可以测试 Markdown 文档是否能正常读取，文本块是否能正常切分，TF-IDF 索引是否能构建，查询接口是否能返回结果，历史记录是否能写入，统计接口是否能正确输出数据。如果每次新增功能后都手动点页面测试，很容易遗漏问题。因此可以逐步引入简单的测试脚本或单元测试，帮助学生理解软件质量保障的基本思想。

项目迭代过程中，最容易出现的问题是新功能破坏旧功能。例如第10次课已经实现了历史记录功能，第11次课增加统计和可视化时，如果强行改变 history.json 的结构，就可能导致原来的历史查询接口失效。因此，在软件工程中，升级功能时要尽量兼容已有数据结构。如果确实需要改变数据格式，也应提供转换逻辑或兼容读取逻辑，而不是让系统中出现新旧格式混合、无法统一处理的情况。

CourseQA 项目虽然规模不大，但已经包含了真实软件项目中的多个重要元素，包括文件读取、文本检索、接口开发、历史记录、日志系统、异常处理、数据统计和可视化展示。通过这个项目，学生不仅能练习 Python 语法，还能理解一个系统如何从单文件脚本逐步演化为模块化工程。更重要的是，学生可以看到每次新增功能都需要考虑原有结构、数据格式、接口调用和测试验证，这正是软件工程课程希望培养的能力。
